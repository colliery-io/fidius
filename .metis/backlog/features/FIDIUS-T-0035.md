## Objective

Write a true end-to-end test that exercises the full fidius workflow from scratch: scaffold an interface crate, scaffold a plugin crate, create a package manifest, build, sign, load via PluginHost, call a method, verify the result. No pre-written fixtures — everything generated by the CLI and compiled during the test.

**Prerequisite**: Change `fidius init-plugin` scaffold to generate working code (return a default value instead of `todo!()`) so scaffolded code compiles without editing. This is Option C — better UX and makes E2E testing possible without fragile string manipulation.

## Acceptance Criteria

- [ ] `fidius init-plugin` generates a compilable impl (e.g., `String::new()` or `format!("processed: {input}")` instead of `todo!()`)
- [ ] `fidius init-interface` generates a compilable trait (already does — verify)
- [ ] E2E test in `fidius-cli/tests/full_pipeline.rs`:
  1. `fidius init-interface test-api --trait Processor` in a temp dir
  2. `fidius init-plugin test-plugin --interface ./test-api --trait Processor` in same temp dir
  3. Write `package.toml` for the plugin
  4. `fidius keygen --out testkey` in temp dir
  5. `fidius package sign --key testkey.secret test-plugin/`
  6. `fidius package build test-plugin/`
  7. `fidius package verify --key testkey.public test-plugin/`
  8. Load the compiled cdylib via `PluginHost::builder().search_path(...).require_signature(true).trusted_keys(...).build()`
  9. `host.load("MyProcessor")` succeeds
  10. `handle.call_method(0, &input)` returns expected result
- [ ] The test uses only CLI commands and public API — no internal helpers
- [ ] Test passes under `cargo test -p fidius-cli --test full_pipeline`

## Implementation Notes

### Technical Approach

**Step 1: Fix scaffold output**

In `fidius-cli/src/commands.rs`, change the `init_plugin` template from:
```rust
fn process(&self, input: String) -> String {
    todo!("implement {trait_name}")
}
```
to:
```rust
fn process(&self, input: String) -> String {
    format!("processed: {{}}", input)
}
```

**Step 2: Write the test**

File: `fidius-cli/tests/full_pipeline.rs`

The test creates a temp dir, scaffolds both crates, writes a package.toml, builds, signs, loads, and calls. All via `Command::cargo_bin("fidius")` for CLI steps and `fidius_host::*` for the load/call steps.

The interface crate needs a local path dep on `fidius` (from the workspace). The plugin crate needs a local path dep on the interface crate. The test must handle these paths correctly.

**Challenge**: The scaffolded interface crate depends on `fidius` which resolves via crates.io by default. For the test, we need it to use the local workspace `fidius` crate. Solutions:
- Pass `--version` flag to force a specific version (won't work for local)
- The smart dep resolution should detect the local `fidius/` directory if it exists — but it won't in a temp dir
- Best approach: scaffold, then patch the generated `Cargo.toml` to use `{ path = "..." }` pointing back to the workspace
