## Bug Description

`#[plugin_impl(TraitName)]` generates code that references bare names from the interface crate:
- `TraitName_OPTIONAL_METHODS` (const array)
- `__fidius_build_traitname_descriptor` (const function)
- `TraitName_VTable` (struct)
- `new_traitname_vtable` (constructor)
- `__FIDIUS_INTERFACE_NAME_TraitName` (const)

When the interface trait and plugin impl are in **separate crates** (the intended architecture), these names are not in scope unless the plugin uses `use interface_crate::*` (glob import).

### Reproduction

1. Create an interface crate with `#[plugin_interface]` on a trait
2. Create a separate plugin crate depending on the interface crate
3. Use explicit imports: `use my_interface::{plugin_impl, MyTrait, PluginError};`
4. Add `#[plugin_impl(MyTrait)]` on the impl block
5. `cargo build` fails with `cannot find value TraitName_OPTIONAL_METHODS in this scope`

### Current Workaround

Use `use my_interface::*` instead of explicit imports. This works but is a design smell — forces glob imports and leaks internal names into the plugin's namespace.

### Root Cause

Two separate issues:

**Issue 1**: `#[plugin_impl]` in `fidius-macro/src/impl_macro.rs` generates code referencing `{Trait}_OPTIONAL_METHODS` and `__fidius_build_{trait}_descriptor` as bare identifiers. These are generated by `#[plugin_interface]` in the interface crate. When the impl is in a different crate, these names must be explicitly imported.

**Issue 2**: The macro generates code referencing `fidius_core::` paths directly (e.g., `fidius_core::wire::serialize`, `fidius_core::status::STATUS_OK`). This requires the plugin crate to have `fidius-core` as a direct dependency, even though the interface crate already depends on `fidius` (which re-exports `fidius-core`).

## Implementation Notes

### Fix for Issue 1: Companion module for generated items

Have `#[plugin_interface]` put all generated items inside a companion module (e.g., `pub mod __fidius_Processor { ... }`). The trait itself stays at crate root. `#[plugin_impl]` references items via qualified paths: `__fidius_Processor::OPTIONAL_METHODS`, `__fidius_Processor::VTable`, etc.

This means plugin authors import the trait by name and the companion module comes along automatically. No glob import needed:

```rust
use my_interface::{plugin_impl, Processor, __fidius_Processor};
// or just: use my_interface::*;  (both work)
```

**Changes required in `fidius-macro/src/interface.rs`**:
- Wrap generated VTable struct, constants, vtable constructor, descriptor builder, and interface name const inside `pub mod __fidius_{TraitName} { ... }`
- Keep the trait definition outside the module

**Changes required in `fidius-macro/src/impl_macro.rs`**:
- Reference all generated items via `__fidius_{TraitName}::` prefix instead of bare names

### Fix for Issue 2: Route through fidius facade

Change the macro to generate `fidius::wire::serialize` instead of `fidius_core::wire::serialize`. Since the interface crate depends on `fidius` and re-exports it, the plugin gets `fidius` transitively. No direct `fidius-core` dep needed.
