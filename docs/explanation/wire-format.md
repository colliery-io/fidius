# Wire Format and Debug/Release Behavior

How fidius serializes data across the FFI boundary, why it uses different
formats in debug and release builds, and what can go wrong.

## The Design Decision

All data crossing the FFI boundary -- method arguments, return values, and
error details -- is serialized via serde. Fidius uses two wire formats:

| Build mode | Wire format | Library | Characteristics |
|------------|-------------|---------|-----------------|
| Debug (`cfg(debug_assertions)`) | JSON | `serde_json` | Human-readable, inspectable, larger, slower |
| Release (`!cfg(debug_assertions)`) | bincode | `bincode` | Compact, fast, binary, not inspectable |

This is controlled at compile time in `fidius-core/src/wire.rs`:

```rust
#[cfg(debug_assertions)]
pub const WIRE_FORMAT: WireFormat = WireFormat::Json;

#[cfg(not(debug_assertions))]
pub const WIRE_FORMAT: WireFormat = WireFormat::Bincode;
```

The `serialize` and `deserialize` functions have two conditional
implementations that delegate to `serde_json` or `bincode` based on the same
`cfg` flag.

## Why JSON in Debug

During development, being able to inspect what crosses the FFI boundary is
invaluable. JSON wire format means you can:

- Print the serialized bytes as a UTF-8 string and read them
- Set a breakpoint in the shim and examine the raw `in_ptr` data
- Use `serde_json::from_slice::<Value>(&bytes)` to parse unknown payloads
- Diff expected vs actual output in test failures

This matters most during plugin development, when the plugin author is
iterating on their implementation and needs to understand what data the host
is sending and what the plugin is returning.

## Why Bincode in Release

In production, the overhead of JSON serialization and deserialization is
unacceptable for performance-sensitive plugin systems:

- **Size.** JSON includes field names, quotes, and formatting. A struct like
  `{ a: 42, b: 7 }` might serialize to 20+ bytes in JSON but only 16 bytes
  in bincode.
- **Speed.** Bincode writes fixed-size integers directly and length-prefixes
  strings. No string escaping, no number-to-text conversion, no whitespace.
- **Allocation.** Bincode produces smaller buffers, reducing allocation
  pressure in the PluginAllocated strategy.

## How Mismatches Are Detected

The `wire_format` field in `PluginDescriptor` records which format the plugin
was compiled with:

```rust
// In the descriptor builder generated by #[plugin_interface]:
wire_format: fidius_core::wire::WIRE_FORMAT as u8,
```

This means a debug-compiled plugin has `wire_format = 0` (Json) and a
release-compiled plugin has `wire_format = 1` (Bincode).

The host checks this at load time via `validate_against_interface()` in
`fidius-host/src/loader.rs`:

```rust
if let Some(wire) = expected_wire {
    if plugin.info.wire_format != wire {
        return Err(LoadError::WireFormatMismatch {
            got: plugin.info.wire_format as u8,
            expected: wire as u8,
        });
    }
}
```

This catches the most common mistake: building the host in release mode but
loading a plugin that was built in debug mode (or vice versa). Without this
check, the host would try to deserialize bincode as JSON (or JSON as bincode),
producing garbage or a deserialization error deep in a method call.

## Why PluginError.details Is Option\<String\>

The `PluginError` type in `fidius-core/src/error.rs` stores its `details`
field as `Option<String>`, not `Option<serde_json::Value>`:

```rust
pub struct PluginError {
    pub code: String,
    pub message: String,
    pub details: Option<String>,  // JSON string, not Value
}
```

This is a deliberate design choice driven by bincode's limitations:

- **bincode cannot `deserialize_any`.** The `serde_json::Value` type relies on
  `Deserializer::deserialize_any()` to inspect the wire format and decide
  whether to produce a string, number, array, or object. Bincode's
  deserializer does not support `deserialize_any` because bincode is not
  self-describing -- it has no type tags in the wire format.

- **Round-trip safety.** If `details` were `Option<Value>`, serializing a
  `PluginError` with bincode would work (bincode can serialize any `Serialize`
  type), but *deserializing* it would fail because bincode cannot reconstruct
  the `Value` enum without type information.

- **The workaround.** `PluginError::with_details()` accepts a
  `serde_json::Value` and calls `.to_string()` to store it as a JSON string.
  `details_value()` parses the string back into a `Value`. This round-trip
  works with both wire formats because both JSON and bincode can serialize and
  deserialize `String`.

```rust
// Creating an error with structured details:
let err = PluginError::with_details(
    "VALIDATION",
    "validation failed",
    serde_json::json!({"field": "name", "reason": "too short"}),
);
// details is Some("{\"field\":\"name\",\"reason\":\"too short\"}")

// Reading it back:
let val: serde_json::Value = err.details_value().unwrap();
// val == json!({"field": "name", "reason": "too short"})
```

## Implications for Testing

Because `cfg(debug_assertions)` controls the wire format, `cargo test` (debug)
and `cargo test --release` exercise **different code paths** through the wire
module. Both should pass. Run both in CI to cover the JSON and bincode paths.
The test suite in `fidius-core/tests/layout_and_roundtrip.rs` uses conditional
compilation to verify the correct format is active in each mode.

## The WireError Type

`fidius-core/src/wire.rs` defines `WireError` as the error type for
serialization and deserialization:

```rust
#[derive(Debug, thiserror::Error)]
pub enum WireError {
    #[error("json wire error: {0}")]
    Json(#[from] serde_json::Error),

    #[error("bincode wire error: {0}")]
    Bincode(#[from] bincode::Error),
}
```

Both variants exist in the enum regardless of build mode, even though only one
is reachable at runtime. This is intentional:

- The type signature of `serialize` and `deserialize` is the same in both
  modes: `Result<..., WireError>`. Callers do not need conditional compilation.
- Error handling code can match on both variants without `cfg` attributes.
- The unreachable variant costs nothing at runtime (it is never constructed).

### What Can Go Wrong

| Error scenario | WireError variant | When it happens |
|----------------|-------------------|-----------------|
| Invalid JSON bytes | `WireError::Json` | Debug build: plugin sends malformed JSON |
| Invalid bincode bytes | `WireError::Bincode` | Release build: plugin sends malformed bincode |
| Type mismatch | Either variant | Host deserializes into wrong type (e.g., `AddOutput` when plugin sent `MulOutput`) |
| Truncated data | Either variant | Buffer was partially written or corrupted |

In the FFI shims, serialization errors are caught and converted to
`STATUS_SERIALIZATION_ERROR` (-2). The host receives this status code and
returns `CallError::Serialization`. No partial or corrupt data reaches the
caller.

## The Full Error Flow

When a plugin method returns `Err(PluginError { ... })`:

```
Plugin side:
  1. Method returns Err(PluginError)
  2. Shim calls wire::serialize(&err)  → Vec<u8>
  3. Shim sets out_ptr/out_len to the serialized error
  4. Shim returns STATUS_PLUGIN_ERROR (-3)

Host side:
  5. PluginHandle::call_method sees status == STATUS_PLUGIN_ERROR
  6. Reads output buffer as slice
  7. wire::deserialize::<PluginError>(slice) → PluginError
  8. Calls free_buffer(out_ptr, out_len)
  9. Returns Err(CallError::Plugin(plugin_error))
```

Both the success path and the error path use the same wire format. The status
code tells the host whether to deserialize the output buffer as the expected
return type or as a `PluginError`.
